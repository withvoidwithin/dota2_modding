# dota2_modding — План реализации и архитектура

Продвинутый аналог moddota: статический сайт (GitHub Pages), который в рантайме у клиента
собирается из JSON-данных. Данные добываются напрямую из движка Dota 2 и обрабатываются в
объекты-классы, пригодные для генерации аннотаций (Panorama TS, EmmyLua).

> Связанные документы: [notes.md](notes.md) — задача проекта, [netcon.md](netcon.md) — справочник
> по netcon/консоли/логам, [commands.txt](commands.txt) — список дамп-команд.

---

## 1. Ключевые решения (зафиксировано)

| Вопрос | Решение |
|---|---|
| Стек сайта | **Vanilla JS, без сборки**. HTML + CSS + ES-модули, `fetch` JSON в рантайме. |
| История изменений | **Свой diff-движок** (не git). Раз в день: новый дамп → diff с предыдущим каноном → запись изменений по версиям. Полные дампы не дублируем — храним актуальные объекты + дельты. |
| Аннотации | Генерируются дампером в `data/`. Способ доставки пользователю **не зафиксирован** — обдумывается (см. §8.1); не приоритет ближайших фаз. |
| Порядок реализации | **Сначала Panorama**: 1) CSS, 2) JS классы и методы. Затем — vscripts. |
| Каденс сбора | Раз в день (Valve часто катит скрытые обновления). |

## 2. Структура репозитория

```
root/
├── data/                  # Итоговый JSON: канон + история + аннотации (публикуется на Pages)
│   ├── manifest.json      # Реестр доменов: путь к данным + layout + версия
│   ├── meta.json          # Текущая версия/патч игры, дата последнего сбора
│   ├── panorama/
│   │   ├── css.json       # CSS-свойства (канон)
│   │   ├── classes.json   # JS классы/методы/свойства/события (канон)
│   │   └── events.json    # Panorama events (канон)
│   ├── vscripts/          # (поздняя фаза) Lua классы, C++ поля, enums, constants ...
│   ├── console/           # cvarlist и т.п.
│   ├── lists/             # modifiers / abilities / items / entities
│   ├── kv/                # KV-схемы и значения
│   ├── history/
│   │   ├── index.json     # Список прогонов: [{date, patch, counts:{added,removed,changed}}]
│   │   └── <date>.json    # Все изменения за конкретный день по всем доменам
│   ├── annotations/
│   │   ├── ts/*.d.ts       # Panorama TypeScript
│   │   ├── lua/*.lua       # EmmyLua
│   │   └── bundles/*.zip   # Архивы для скачивания (готовит дампер)
│   └── search-index.json  # Предвычисленный индекс для глобального поиска
│
├── api_dumper/            # Node.js: сбор + обработка + diff + генерация (см. §4)
├── dota_addon/            # Аддон Dota 2: рефлексия API изнутри игры (см. §5)
├── dota2_modding/         # Статический сайт (см. §6)
│
├── config.js              # Корневые структурные константы (пути к data/, manifest)
├── notes.md  netcon.md  commands.txt  plan.md
```

**Источник истины — `data/`** (один экземпляр, без дублирования). Сайт и аддон не пишут в него
напрямую; пишет только дампер.

## 3. Конвейер данных (сквозной поток)

```
[Dota 2 + аддон]  →  [api_dumper: сбор]  →  [нормализация в объекты-классы]
                                                   │
                                                   ▼
                            [diff с предыдущим каноном]  →  history/<date>.json
                                                   │
                                                   ▼
                     [обновление канона: firstSeen/lastSeen/removed]  →  data/*.json
                                                   │
                                                   ▼
                       [генерация аннотаций + search-index + zip]  →  data/annotations, search-index
                                                   │
                                                   ▼
                         [commit/push]  →  GitHub Actions  →  GitHub Pages (сайт + data)
```

Два канала сбора:
- **netcon (TCP)** — консольные дамп-команды (`dump_panorama_css_properties`, `cvarlist`,
  `dump_modifier_list`, `dumpstringtable ...`). Захват по маркерам `echo BEGIN/END` (см. netcon.md §7).
- **HTTP от аддона** — то, чего нет в консольных командах: полное дерево Lua-классов/методов и
  Panorama JS API. Аддон делает рефлексию изнутри игры и POST'ит JSON на приёмник дампера.

## 4. api_dumper — архитектура

Node.js. За основу берётся проверенный дизайн прежнего прототипа (express + netcon + модули дампа +
дашборд), расширенный слоями нормализации, diff и генерации.

```
api_dumper/
├── .env                       # Машинно-специфично: DOTA_PATH, DOTA_ADDON_NAME, NETCON_PORT, HTTP_PORT, теги
├── package.json               # scripts: init / dev ; deps: express, dotenv
├── dashboard/
│   ├── index.html  style.css  app.js     # UI: запуск, мониторинг, кнопки дампов, "собрать всё", commit/push
│   └── core/
│       ├── server.js          # Express: статика дашборда, HTTP-приёмник аддона, роуты дампов, SSE
│       ├── launcher.js        # Запуск dota2.exe + vconsole2.exe с флагами (-dev -tools -insecure -console -netconport -addon)
│       ├── processes.js       # Проверка запущенных процессов (tasklist) — защита от двойного старта
│       ├── netcon.js          # TCP-клиент: буферизация по \n, переподключение, события, send()
│       ├── lib/
│       │   ├── emitter.js     # Общий синхронный pub/sub
│       │   └── env.js         # Загрузка+валидация .env, типизированный конфиг
│       ├── dumps/             # Один файл = один netcon-дамп: { name, command, parser, domain, outputPath, layout }
│       ├── receivers/         # Обработчики HTTP-пейлоадов аддона (panorama-js, vscript-lua)
│       ├── model/             # Нормализация сырого вывода → объекты-классы (по доменам)
│       ├── history/
│       │   └── diff.js        # Diff-движок + запись history/<date>.json + обновление firstSeen/lastSeen/removed
│       └── generate/
│           ├── panorama-ts.js # Канон → *.d.ts
│           ├── emmylua.js     # Канон → *.lua аннотации
│           ├── search-index.js# Канон → search-index.json
│           └── bundles.js     # Архивы аннотаций (zip) для скачивания
```

### Принципы (перенос из прежнего context.md, остаются в силе)
- Стили только в `style.css`, клиентский JS дашборда только в `app.js`.
- Серверная логика разбита по модулям; ничего не пишем монолитом в `server.js`.
- Каждый дамп — отдельный файл в `dumps/`; добавление дампа не требует правок HTML.
- Все машинно-специфичные настройки только через `.env`; структурные константы — в корневом `config.js`.
- netcon буферизует частичные строки между TCP-чанками; захват ответа команды — по уникальным
  маркерам `echo <TAG> DUMP_START/END:<module>` с таймаутом и сбросом сессии при дисконнекте.
- Одновременно — один активный дамп (блокировка сессией).
- Защита от двойного запуска игры: `checkProcesses` перед `launch`.

### Оркестрация «собрать всё» (для ежедневного прогона)
Одна команда последовательно: запуск игры → ожидание netcon → прогон всех `dumps/` + сбор HTTP от
аддона → нормализация → diff → запись канона+истории → генерация аннотаций/индекса/zip. Идемпотентна,
с таймаутами на каждом шаге.

## 5. dota_addon — архитектура

Аддон, смонтированный в Dota 2 через Workshop Tools; разворачивается junction-ссылками
([setup_symlinks.js](dota_addon/setup_symlinks.js)). Задача — рефлексия API изнутри движка и отправка
данных дамперу.

```
dota_addon/
├── game/
│   ├── addoninfo.txt
│   ├── scripts/vscripts/
│   │   └── addon_game_mode.lua    # Bootstrap; (поздняя фаза) рефлексия Lua-классов/методов → HTTP POST дамперу
│   └── panorama/                  # (фаза Panorama JS) layout+JS: интроспекция Panorama API → HTTP POST
│       └── layout + scripts       # перечисление типов панелей, методов (по прототипам), $-API, enums, events
├── content/maps/dump.vmap
└── setup_symlinks.js
```

- **Panorama CSS (первый шаг)** идёт через netcon-команду `dump_panorama_css_properties` — аддон для
  этого не нужен, кроме как для запуска кастомки/окружения.
- **Panorama JS API** — Panorama-скрипт аддона интроспектирует прототипы классов панелей и глобальные
  API, формирует JSON и POST'ит на HTTP-приёмник дампера. (`panorama_dump_symbols` — вспомогательно.)
- **Vscript Lua** (поздняя фаза) — `addon_game_mode.lua` обходит `_G`, метатаблицы классов и FDesc,
  отделяет серверные/клиентские методы и C++ поля, POST'ит дамперу.
- Отправка наружу — только на `127.0.0.1:<HTTP_PORT>` локального дампера (см. netcon.md: порт не
  выставлять наружу).

## 6. dota2_modding — архитектура сайта

Vanilla, без сборки. Рантайм-рендеринг из `data/`.

```
dota2_modding/
├── index.html          # Оболочка приложения
├── app.js              # Хеш-роутер, загрузка manifest.json, диспетчеризация по layout
├── style.css
├── layouts/            # Рендереры по доменам (panorama-css, panorama-classes, vscript-tree, console, ...)
├── components/         # Переиспользуемые блоки: дерево (extends), список, деталь, поиск, таймлайн истории
└── pages/              # Статические разделы: docs/гайды, downloads
```

### Разделы UI (соответствие 9 пунктам задачи из notes.md)
1. **Vscripts** — дерево Lua-классов по `extends`, методы (сервер/клиент), C++ поля. *(поздняя фаза)*
2. **Panorama классы** — дерево/список классов и методов. *(фаза 4)*
3. **Типы панелей** — Panel, Button, DOTAScenePanel и т.д. *(фаза 4–5)*
4. **CSS** — свойства и значения. *(фаза 2, первый домен)*
5. **Console commands** — cvarlist. *(поздняя фаза)*
6. **Events / constants / enums / lists / KV** — panorama+vscript events, константы, энамы, списки
   модификаторов/абилок/предметов/сущностей, KV-схемы. *(фазы 5, 8)*
7. **История изменений** — отдельная страница «что изменилось в конкретный день» по всем данным +
   на карточке каждой сущности: когда появилась / когда удалена. *(фаза 3)*
8. **Глобальный поиск** — по предвычисленному `search-index.json`, в памяти клиента. *(фаза 10)*
9. **Документация и гайды** — авторские страницы. *(фаза 11)*
10. **Downloads** — скачивание сгенерированных аннотаций (TS/Lua/zip). *(фаза 6, 9)*

### Загрузка данных
- `manifest.json` описывает домены: `{ label, dataPath, layout, version }`.
- Крупные дампы (напр. cvarlist ~44k строк) грузятся лениво — по разделу, не на старте.
- Поиск работает по компактному `search-index.json`, а полные данные домена подгружаются при переходе.

## 7. Схема данных и истории

### Канонический объект (любая сущность)
Сущности хранятся как объекты-классы (а не сырые строки), чтобы из них генерировать аннотации и diff.
Каждая несёт метку жизненного цикла:

```jsonc
{
  "id": "panorama.css.flow-children",          // стабильный ключ для diff
  "kind": "css-property",
  "name": "flow-children",
  // ...доменные поля (values, syntax, params, returns, extends, flags, description)
  "firstSeen": "2026-06-24",                    // дата/патч появления
  "lastSeen":  "2026-06-24",                    // дата последнего наблюдения
  "removed":   null                              // или дата удаления, если исчезла из дампа
}
```

Канон всегда содержит **актуальное состояние на сейчас** + удалённые сущности с пометкой `removed`
(чтобы сайт показывал «удалено такого-то числа»).

### История за день
```jsonc
// data/history/<date>.json
{
  "date": "2026-06-24",
  "patch": "7.39c",
  "changes": [
    { "domain": "panorama/css", "op": "added",   "id": "...", "after": { } },
    { "domain": "panorama/css", "op": "removed", "id": "...", "before": { } },
    { "domain": "panorama/css", "op": "changed", "id": "...", "before": { }, "after": { }, "fields": ["values"] }
  ]
}
```

### Diff-движок (собственный, не git)
1. Загрузить предыдущий канон домена.
2. Сопоставить по `id`: новые → `added` (`firstSeen = today`); исчезнувшие → `removed`
   (`removed = today`, объект сохраняется); общие — сравнить поля → `changed` при отличиях,
   `lastSeen = today`.
3. Записать дельты в `history/<date>.json`, обновить `history/index.json`, перезаписать канон.
- Diff детерминирован, поля сравниваются по стабильному порядку; объём истории = только дельты.

## 8. Публикация (GitHub Pages)

Сайт и `data/` лежат в разных папках, но Pages должен отдавать их вместе. Деплой через **GitHub
Actions**: workflow собирает артефакт `dota2_modding/` + `data/` (+ `.nojekyll`) и публикует на Pages.
Сам сайт остаётся без сборки/бандлера — Actions только склеивает и выкладывает. Из дашборда дампера —
кнопка commit/push для запуска прогона публикации.

### 8.1. Доставка аннотаций пользователю — ОТКРЫТЫЙ ВОПРОС

Способ, которым конечный пользователь получает сгенерированные аннотации с сайта, **ещё
обдумывается** — решение отложено, в ближайших фазах не реализуется. Рассмотренные варианты (на
будущее):

1. Скачивание отдельных файлов (по домену/классу).
2. Скачивание архивом (zip-бандл).
3. Copy-to-clipboard / инлайн прямо на карточке класса/метода.
4. npm-пакет + GitHub Release (авто-апдейт).
5. Прямой URL / CDN (jsDelivr, raw Pages).
6. Git submodule репозитория.
7. Установщик / CLI (`npx`, кладёт типы в проект + правит `.luarc.json`).
8. VSCode-расширение / LuaLS-addon.

Генерация аннотаций в `data/` остаётся в плане (фазы 6, 9); выбор канала доставки делаем позже.

## 10. Конвенции и правила

- Если не хватает данных для решения — **спросить, не придумывать**.
- Каждый модуль — отдельный файл; ничего монолитного.
- Пишет в `data/` только дампер; сайт и аддон — только читают / отправляют.
- Все доменные данные — объекты-классы со стабильным `id` (требование diff и генерации аннотаций).
- Стили — только в `style.css`; машинные настройки — только в `.env`; структурные пути — в `config.js`.
- После завершённого этапа — актуализировать этот план.
