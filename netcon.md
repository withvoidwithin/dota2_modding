# Dota 2: netcon, консоль и логи — справочник

Самостоятельный технический справочник: как запускать Dota 2 в dev/tools-режиме, подключаться
к движку по netcon (TCP), выполнять консольные команды и получать логи. Не привязан к какому-либо
конкретному проекту — общая информация о работе с движком Source 2 (Dota 2) через консоль.

> Точность: флаги запуска и протокол netcon стабильны и проверены на практике. Доступность
> отдельных консольных команд зависит от сборки игры и режима (главное меню / матч / tools), —
> такие места помечены. Проверяй `find`/`help` в самой консоли.

---

## 1. Термины

- **Консоль (developer console)** — встроенная командная строка движка. Включается флагом `-console`.
- **VConsole (`vconsole2.exe`)** — отдельное GUI-приложение Source 2 для просмотра логов и ввода
  команд; подключается к запущенному инстансу движка.
- **netcon** — сетевой консольный интерфейс: движок открывает **TCP-порт**, к которому можно
  подключиться обычным сокетом и работать с консолью программно (как telnet). Открывается флагом
  `-netconport <port>`. Это «программный эквивалент» VConsole.
- **tools-режим** (`-tools`) — запуск редакторов Source 2 (Asset Browser, Hammer, Particle Editor
  и т.п.). Часть отладочных/дамп-команд доступна именно здесь.

---

## 2. Процессы и пути

В стандартной установке Steam:

```
<Steam>/steamapps/common/dota 2 beta/game/bin/win64/dota2.exe
<Steam>/steamapps/common/dota 2 beta/game/bin/win64/vconsole2.exe
```

- `dota2.exe` — сам клиент/движок.
- `vconsole2.exe` — GUI-консоль (можно запускать отдельно, она подключится к работающему движку).

---

## 3. Флаги запуска dota2.exe

Полезные для консоли/инструментов/netcon:

| Флаг | Назначение |
|---|---|
| `-console` | Включить developer-консоль |
| `-dev` | Режим разработчика (открывает dev-команды/cvar'ы) |
| `-tools` | Запуск в режиме инструментов Source 2 |
| `-insecure` | Отключить VAC; нужно для dev/cheat/custom-сценариев |
| `-netconport <port>` | Открыть TCP-порт netcon (напр. `-netconport 2121`) |
| `-addon <name>` | Смонтировать аддон по имени |
| `-novid` | Пропустить интро |
| `-condebug` | Дублировать вывод консоли в файл `console.log` в каталоге game |
| `-vconsole` | Авто-открыть/подключить VConsole (зависит от сборки) |
| `-windowed -w <W> -h <H>` | Оконный режим и размер (удобно при автоматизации) |

Типичный набор для отладки/дампа:

```
dota2.exe -dev -tools -insecure -console -novid -netconport 2121 -addon <addon_name>
```

> `-netconport` открывает TCP-сервер независимо от того, запущена ли VConsole. VConsole — это
> альтернативный (GUI) способ работать с той же консолью; для программного доступа достаточно netcon.

---

## 4. Запуск игры и консоли

1. **Игра**: запустить `dota2.exe` с нужными флагами (см. выше). Процесс появляется в системе
   сразу, но движок инициализируется несколько секунд — **netcon-порт открывается с задержкой**.
2. **VConsole (опционально, GUI)**: запустить `vconsole2.exe` без аргументов — подключится к
   работающему инстансу. Для программного сбора логов VConsole не обязателен.
3. **Custom game / карта** (когда движок готов): через консоль/netcon, напр.
   `dota_launch_custom_game <addon_name> <map_name>` или `map <mapname>`.

Запуск процессов «откреплённо» (detached), чтобы не блокировать управляющий скрипт — стандартная
практика (`spawn` с `detached`, `CreateProcess` и т.п.).

---

## 5. Протокол netcon (TCP)

netcon — **сырой текстовый поток поверх TCP**, без обрамления сообщений.

### Подключение
- Адрес: `127.0.0.1`, порт: тот, что задан в `-netconport`.
- Подключаться можно только после того, как движок **откроет порт** (несколько секунд после
  старта процесса). До этого `connect` будет отвергнут — нужен **повтор с интервалом** (например,
  раз в 1–2 секунды) и авто-переподключение при разрыве.
- Это локальный отладочный интерфейс — не выставляй порт наружу. Некоторые сборки поддерживают
  `-netconpassword <pw>`; тогда **первая отправленная строка** должна быть паролем.

### Отправка команд
- Команда — это строка, завершённая **`\n`** (перевод строки). Пример: `echo hello\n`.
- Всё, что отправлено, исполняется движком как консольная команда.
- Можно слать несколько команд подряд (каждую со своим `\n`). Они исполняются по порядку.

### Чтение вывода
- Вывод приходит как **UTF-8 текст**, построчно, но **TCP не сохраняет границы строк/сообщений**:
  одна логическая строка может прийти двумя пакетами, а несколько строк — одним. **Буферизуй
  входящие данные и режь по `\n`**, сохраняя «хвост» (неполную строку) до следующего чтения.
- Поток содержит **весь** консольный вывод движка (спам подсистем), а не только ответ на твою
  команду. Ответы приходят **асинхронно** и **перемешаны** с прочими сообщениями.
- Эхо введённой команды обратно — **не гарантировано** (зависит от настроек). Не полагайся на него.
- Большие дампы (тысячи строк) приходят множеством TCP-пакетов — читай в цикле до конца.

### Переподключение
- На `close`/`error` сокета: сбросить буфер, занулить сокет, повторить `connect` через интервал.
- Порт может пропасть при выходе из игры/смене состояния — переподключение должно быть постоянным.

## 6. Примеры подключения

### Node.js
```js
const net = require("net");

const PORT = 2121;
let buffer = "";

const sock = net.connect(PORT, "127.0.0.1", () => {
    sock.write("echo connected\n");
    sock.write("cvarlist\n");
});

sock.on("data", (chunk) => {
    buffer += chunk.toString("utf8");
    const lines = buffer.split("\n");
    buffer = lines.pop();               // сохранить неполную строку
    for (const line of lines) console.log(line);
});

sock.on("error", () => { /* порт ещё не открыт или разрыв → повторить connect */ });
sock.on("close", () => { /* запланировать переподключение */ });
```

### Python
```python
import socket

PORT = 2121
s = socket.create_connection(("127.0.0.1", PORT))
s.sendall(b"echo connected\n")
s.sendall(b"cvarlist\n")

buf = b""
while True:
    chunk = s.recv(4096)
    if not chunk:
        break
    buf += chunk
    *lines, buf = buf.split(b"\n")      # buf хранит неполную строку
    for line in lines:
        print(line.decode("utf-8", "replace"))
```

## 7. Захват вывода конкретной команды (паттерн маркеров)

Поскольку вывод асинхронный и перемешан с прочим спамом, надёжный способ выделить ответ команды —
**обрамить её уникальными маркерами через `echo`**:

```
echo <UNIQUE_BEGIN>
<нужная команда>
echo <UNIQUE_END>
```

Алгоритм на стороне клиента:
1. Отправить три строки выше (каждую с `\n`).
2. Читать поток построчно.
3. Начать собирать строки **после** появления `<UNIQUE_BEGIN>`.
4. Прекратить и считать результат готовым при появлении `<UNIQUE_END>`.

Рекомендации:
- Маркер делать редким/уникальным (префикс-тег + имя операции), чтобы он не совпал со строкой
  самого дампа. Сравнивать строки на точное вхождение маркера.
- Предусмотреть **таймаут** на случай, если `<UNIQUE_END>` не пришёл (движок завис/отключился).
- Команды исполняются по порядку, поэтому END придёт после полного вывода команды.

---

## 8. Полезные консольные команды

### Общие (движок Source 2)
| Команда | Назначение |
|---|---|
| `echo <text>` | Напечатать строку в консоль (основа маркеров) |
| `find <substr>` | Найти cvar'ы/команды по подстроке |
| `help <name>` | Справка по cvar/команде |
| `cvarlist` | Список всех cvar'ов (имя, значение, флаги, описание) |
| `exec <file.cfg>` | Выполнить конфиг из каталога cfg |
| `developer <0..2>` | Уровень dev-вывода |
| `clear` | Очистить консоль |
| `status` | Текущий статус сервера/сессии |
| `disconnect` / `quit` | Отключиться / выйти |
| `sv_cheats 1` | Включить чит-команды (многие dev-команды под этим гейтом) |
| `map <mapname>` | Загрузить карту |

### Логи в файл
| Команда | Назначение |
|---|---|
| `con_logfile <path>` | Начать писать консоль в файл; `con_logfile ""` — остановить |
| `condump` | Сбросить текущий буфер консоли в файл `condumpNNN.txt` (каталог game) |

### Фильтрация шума (если поддерживается сборкой)
| Команда | Назначение |
|---|---|
| `con_filter_enable <0..2>` | Включить фильтр вывода консоли |
| `con_filter_text <substr>` | Показывать только строки с подстрокой |

### Custom games / аддоны
| Команда | Назначение |
|---|---|
| `dota_launch_custom_game <addon> <map>` | Запустить кастомную игру (нужен `-addon`, `-insecure`) |

### Семейство дамп-команд Dota (доступность зависит от сборки/режима)
Проверяй наличие через `find dump` / `find dota_dump`. Известные кандидаты:
```
cvarlist
dump_panorama_css_properties
dump_panorama_events
dump_modifier_list
panorama_dump_symbols 0|1
cl_dump_response_symbols
dump_response_symbols
dumpstringtable <table> sv verbose      # напр. ModifierNames, LuaModifiers, EconItems, Scenes ...
cl_dumpentity <ent_index>               # рантайм, в матче
cl_showents                             # рантайм
```
> Многие из них требуют `-dev`/`-tools`/`sv_cheats` и/или конкретного состояния (главное меню vs
> загруженная карта vs tools). Если команда «молчит» — почти всегда дело в режиме/гейте.

## 9. Способы получения логов (сводно)

1. **netcon (реальное время)** — подключиться по TCP и читать поток (см. §5–7). Лучший вариант для
   автоматизации: двусторонний (команды + вывод), без файлов.
2. **VConsole** — GUI: смотреть/фильтровать логи и вводить команды вручную.
3. **`-condebug`** — движок пишет весь вывод в `console.log` в каталоге game (читать файл).
4. **`con_logfile <path>` … `con_logfile ""`** — программно включить/выключить запись в свой файл
   вокруг нужных команд.
5. **`condump`** — разовый сброс текущего буфера консоли в файл.

---

## 10. Состояния и гейты (почему команда не отвечает)

- **Движок ещё не готов.** Сразу после старта процесса консоль/порт недоступны несколько секунд.
- **Не тот режим.** Часть команд только в `-tools`, часть только в загруженной карте/матче, часть —
  в главном меню. Дамп-команды ассетов/таблиц часто требуют tools или активной сессии.
- **`sv_cheats`/`-dev`.** Многие отладочные команды скрыты, пока не включены чит/дев-режимы.
- **Аддон не смонтирован.** `dota_launch_custom_game` и аддон-специфика требуют `-addon` и `-insecure`.
- **Слишком рано отправлено.** Команда, посланная до полной инициализации, может потеряться.

---

## 11. Заметки о формате вывода

- Кодировка — **UTF-8**.
- Вывод **построчный**, но границы по TCP не сохраняются (буферизуй, режь по `\n`).
- Поток **смешанный**: ответ команды перемешан со спамом подсистем → выделяй маркерами.
- Движок может **префиксовать** строки тегами подсистем; формат строк команды задаёт сама команда
  (CSV-подобный, таблицы, markdown-подобный — у разных дампов по-разному), парсить нужно под каждую.
- Порядок исполнения команд сохраняется (FIFO), поэтому маркер END достоверно завершает блок.

## 12. Траблшутинг

| Симптом | Причина / решение |
|---|---|
| `connect` отвергается | Порт ещё не открыт (движок грузится) или игра не с `-netconport`. Повтор с интервалом. |
| Подключились, но тишина | Команда отправлена без `\n`; либо команда требует другого режима/`sv_cheats`. |
| Строки «рвутся» / склеиваются | Не буферизуется TCP-поток. Накапливай и режь по `\n`, храни хвост. |
| Не видно ответа конкретной команды | Вывод перемешан — используй echo-маркеры BEGIN/END. |
| Команда «не существует» | Зависит от сборки/режима. Проверь `find <substr>` и режим (`-tools`/карта). |
| Дамп обрывается | Большой вывод идёт многими пакетами — читай в цикле; не закрывай сокет рано. |
| Кириллица/символы битые | Декодируй как UTF-8 (с заменой невалидных байт). |

## 13. Шпаргалка

```
# Запуск (отладка/дамп)
dota2.exe -dev -tools -insecure -console -novid -netconport 2121 -addon <addon>

# Подключение
TCP → 127.0.0.1:2121   (повтор connect, пока порт не откроется)

# Команда
<текст>\n

# Чтение
буферизовать поток, резать по \n, хранить неполный хвост

# Захват ответа команды
echo <BEGIN>\n   <command>\n   echo <END>\n
→ собирать строки между BEGIN и END (+ таймаут)

# Логи в файл
con_logfile out.txt\n  <commands>  con_logfile ""\n
# или -condebug → console.log

# Кастомка
dota_launch_custom_game <addon> <map>\n
```
