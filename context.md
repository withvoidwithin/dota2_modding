# Dota 2 API Dumper — Context

## Что это
Локальный инструмент для дампа API Dota 2. Состоит из Node.js дашборда и аддона для Dota 2.
Дашборд — локальный сайт, запускаемый через `npm start` в VSCode.

## Репозиторий
```
DOTA2_MODDING/                          # Корень репозитория
├── config.js                           # Корневые константы проекта (DATA_DIR)
├── data/                               # Дампы в JSON (результат работы модулей)
│   └── modifier_list/
│       └── data.json
├── api_dumper/                         # Node.js дашборд
│   ├── dashboard/
│   │   ├── core/
│   │   │   ├── server.js               # Express сервер
│   │   │   ├── processes.js            # Проверка запущенных процессов через tasklist
│   │   │   ├── launcher.js             # Запуск dota2.exe и vconsole2.exe
│   │   │   ├── netcon.js               # TCP-клиент netcon, SSE-события, отправка команд
│   │   │   ├── dump_manager.js         # Менеджер сессий дампа (маркеры, сбор строк, сохранение)
│   │   │   └── dumps/
│   │   │       └── modifier_list.js    # Модуль дампа modifier_list
│   │   ├── index.html                  # UI дашборда
│   │   ├── style.css                   # Стили
│   │   └── app.js                      # Клиентский JS (polling, SSE, обновление UI)
│   ├── .env                            # Переменные окружения
│   ├── package.json
│   └── package-lock.json
└── dota_addon/                         # Аддон Dota 2 (Lua + Panorama)
    ├── content/
    ├── game/
    └── setup_symlinks.js
```

## Зависимости
- `express` — сервер
- `dotenv` — чтение .env
- `child_process`, `path`, `fs` — встроенные Node.js

## Как запустить
```bash
cd api_dumper
npm run init  # node dashboard/core/server.js
npm run dev   # node --watch dashboard/core/server.js
```

## Разработка
### ✅ Этап 1 — Инфраструктура дашборда
Express сервер, раздача статики, структура файлов, .env.
Роутинг в server.js — подключение модулей
Базовый UI с состояниями кнопок (disabled/enabled в зависимости от статуса процессов)

### ✅ Этап 2 — Мониторинг процессов
Модуль проверки запущен ли dota2.exe и vconsole
Кнопка запуска Dota 2 с параметрами (-console, -novid и т.д.)
Статус отображается в дашборде

### ✅ Этап 3 — Netcon
Модуль подключения к netcon по TCP (netcon.js)
Авто-коннект при запуске сервера, переподключение каждые 2 секунды при обрыве
Статус netcon (connected/disconnected) добавлен в /api/processes
SSE-поток /api/netcon/stream — стриминг логов на клиент в реальном времени
Эндпоинт POST /api/netcon/command — отправка команд в консоль Dota
Лог-панель и поле ввода команд в UI дашборда

### ✅ Этап 4 — Модули дампа
Универсальный dump_manager.js — сессия дампа, маркеры START/END, сбор строк, сохранение JSON
Первый модуль: modifier_list (команда dump_modifier_list, результат отсортирован по алфавиту)
Правая панель в UI с кнопками модулей, блокируется если netcon офлайн
Кнопка показывает состояние dumping во время сбора данных
dump_end событие через SSE — клиент получает уведомление о завершении
Эндпоинт GET /api/config — проброс нужных env-переменных на клиент
Цветовая маркировка кастомных логов по тегу и сабтегу

### Этап 5 — HTTP приёмник
Эндпоинт в server.js для приёма данных с Lua сервера аддона
Сохранение в JSON

### Этап 6 — Генерация
Модуль генерации EmmyLua аннотаций из JSON
Модуль генерации Panorama TS из JSON
Модуль генерации статического сайта

### Этап 7 — Публикация
Настройка GitHub Pages
Кнопка коммита и пуша

## Важные детали
- Для работы netcon нужны оба процесса: Dota 2 И VConsole
- Модули дампа — отдельные JS файлы в dashboard/core/dumps/, в UI это правая панель с кнопками
- Кнопки модулей блокируются если netcon не подключён
- Сбор данных двумя способами: через netcon (TCP) и через HTTP с Lua аддона
- Дампы сохраняются в DOTA2_MODDING/data/<module_name>/data.json

### Правила разработки
- Если не хватает данных — спросить, не придумывать
- Двигаться небольшими шагами
- JS логика только в .js файлах, не в .html
- Каждый модуль — отдельный файл, не писать всё в server.js
- После выполненого этапа нужно актуализировать context.md

### Соглашения по коду
- Стили только в style.css
- Клиентский JS только в app.js
- Серверная логика разбита по модулям: processes.js, launcher.js, netcon.js и т.д.
- Все машинные настройки только через .env, не хардкодить

### Важные решения принятые в процессе
- server.js лежит в dashboard/core/, а не в корне api_dumper/
- Кнопка запуска запускает оба процесса сразу (Dota + VConsole)
- Защита от двойного запуска на уровне сервера (checkProcesses перед launch)
- Polling каждую секунду для статусов процессов
- SSE для стриминга netcon-логов (вместо polling)
- Netcon авто-коннектится при старте сервера, не ждёт команды от клиента
- Маркеры дампа: echo с тегом до и после команды — `@API_DUMPER:DASHBOARD DUMP_START:<module>`
- Одновременно может выполняться только один дамп (блокировка через session в dump_manager)
- Конфиг для клиента через /api/config (env-переменные не хардкодятся в app.js)
- Цветовая маркировка логов: @API_DUMPER → #9881FF, :LUA → #98C8FF, :DASHBOARD → #FFE9A2
- SSE и polling инициализируются в init(), которая вызывается после загрузки /api/config — защита от race condition
- config.js в корне репозитория — source of truth для структурных констант общих между модулями (DATA_DIR)
- .env содержит только машинно-специфичные и api_dumper-специфичные настройки
- dump_manager.js берёт DATA_DIR из корневого config.js, а не из .env
- Функция обработки строк в модулях дампа называется parseLines (экспортируется под ключом process)
