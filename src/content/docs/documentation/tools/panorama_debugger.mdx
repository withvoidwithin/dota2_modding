## 1. Определение
**Panorama Debugger** — это интегрированная среда отладки в виде отдельного окна, позволяющая просматривать, изменять и диагностировать интерфейс игры в реальном времени.    
**Panel** (панель) — это элемент интерфейса. Может быть кнопкой, изображением, лейблом или другим специфическим UI элементом **Dota 2**, например **DOTAHeroImage**.

## 2. Использование

**Panorama Debugger** можно открыть с помощью клавиши **F6** с включенным режимом **-dev** или при запуске через **Dota 2 Workshop Tools**.

Окно дебаггера разделено на несколько основных областей:
![Panorama Debugger Layout](@assets/documentation/panorama_debugger/layout.png)

### [A] Панель управления
При нажатии на **F6** **Panorama Debugger** открывается и сразу же сворачивается, включая режим выделения UI панели с помощью курсора.
После выбора панели в окне игры дебаггер снова разворачивается и подсвечивает ее в иерархии (область B).

#### Кнопки управления

* **Просмотреть** — Кнопка ручного включения режима выделения UI панели в окне игры.
* **Фильтр по имени** — Вписав в поле текст, вы отфильтруете иерархию (B), чтобы показывать только те панели, которые содержат указанный текст по типу панели, ID или значению любого атрибута.
* **Размер дебаггера** — С помощью кнопок **Меньше** или **Больше** можно отрегулировать размер окна дебаггера вместе с размером шрифта. Сбросить до оригинального размера можно с помощью кнопки **Ориг. размер**.
* **Слайдер прозрачности** — Регулирует прозрачность подсветки выделенной панели в окне игры.
* **Отмена изменений** — Кнопка **Отменить все изменения** сбрасывает все измененные CSS свойства, указанные в области (C), к их оригинальным значениям.

##### Данные для разработчиков
Кнопка включения дополнительных служебных отладочных данных панелей движка Source 2. Например, адрес в памяти C++ объекта.
На практике для разработки кастомок это бесполезно, ведь нет прямого доступа к низкоуровневой логике и памяти игры.

##### Данные об окраске
Кнопка, включающая режим отладки рендера UI панелей в области иерархии (B).
По каждой UI панели показывается информация о ее статусе отрисовки.
![Panorama Debugger Layout](@assets/documentation/panorama_debugger/panel_paint_data.png)

Красный цвет показывает статус отрисовки панели: если цвет постоянно горит, значит панель обрабатывается каждый кадр.
Если красный цвет постепенно угасает, значит панель была обработана один раз.
Зеленая рамка показывает наличие свойства прозрачности у панели.

###### R (Rules)
**Нагрузка**: низкая    
Показывает статус обработки правил панели. Если панель видимая, то каждый кадр она проверяет различные правила. Например, находится ли на ней курсор, чтобы изменить стиль `:hover` и вызвать JS ивент.
Если данные об отладке показывают, что R постоянно красная — это нормально, когда она является видимой.

###### S (Style)
**Нагрузка**: средняя    
Показывает статус обновления стилей панели.
В обычной практике вполне нормально, когда панель периодически получает новые стили. В основном это связано с интерактивным взаимодействием, например, при наведении курсора или клике по элементу.
Если же панель постоянно получает новые стили за очень короткий промежуток времени, то это может указывать на неоптимальную логику в JS скриптах.

###### P (Paint/Render)
**Нагрузка**: высокая    
Показывает статус рендера панели. Обычно все статичные панели рендерятся 1 раз и кешируются движком, чтобы не пересчитывать отрисовку каждый кадр.
Все панели, которые имеют анимации, изменяющие любое визуальное свойство, будут триггерить перерисовку. Webm, партиклы и CSS keyframe анимации будут постоянно держать этот статус в красном цвете и это нормально.

###### L (Layout)
**Нагрузка**: Самая высокая    
Показывает статус обновления компоновки панели.
Каждый раз, когда панель изменяет свои размеры или позицию, движок должен пересчитать компоновку ее, всех дочерних панелей и, возможно, даже родительских, а затем вызвать еще и их рендер(L).
Это самая тяжелая операция из всех перечисленных. Если вы видите, что (L) постоянно красная на панелях, которые не должны изменяться, то это обычно указывает на неоптимальную логику в JS скриптах или верстке самих панелей.

### [B] Иерархия панелей
Иерархическое дерево всех UI панелей, загруженных в текущий момент в память игры.
При выборе любой панели в дереве в той же строке в иерархии появится название XML файла, из которого была создана панель. А в области (C) отобразятся все свойства выбранной панели.
В самих панелях отображаются далеко не все атрибуты, которые фактически у них установлены.

### [C] Свойства выбранного элемента
Каждая панель обладает стандартными CSS свойствами.

#### Стиль
На вкладке **Стиль** отображаются все нестандартные CSS свойства выбранной панели.
В этом поле можно изменять любые CSS свойства панели в реальном времени или добавлять новые.
Учтите, что, изменяя свойства класса, вы также измените их глобально для всех панелей, которые его используют.
Из-за этого бывает, что кнопка **Отменить все изменения** не срабатывает корректно, после чего остается только полностью перезапускать игру.

#### Вычислено
Это вкладка, которая показывает все итоговые фактические CSS и некоторые другие специфические свойства выбранной панели.

#### Переменные и Атрибуты
Вкладки для отображения переменных и кастомных атрибутов выделенной панели.

### [D] Скрипты и консоль
В текущей версии поле консоли работает **некорректно**.
Раньше там можно было вызывать JS функции и просто выполнять JS код в контексте выделенной панели с помощью метода `$.GetContextPanel()`.

## 3. Кастомизация
Существует возможность поменять тему оформления **Panorama Debugger** с помощью консольной команды `panorama_debugger_theme <theme_name>`.
В наличии есть 2 базовые темы **Dark** и **Light**.
После выполнения команды необходимо перезапустить дебаггер.

Кроме этого существует возможность оверайднуть стандартный `...\panorama\styles\debugger.css` файл, отвечающий за стили дебаггера, и даже написать свою тему.
