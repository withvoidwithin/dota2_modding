## 1. Определение

**KeyValues** (KV) — это формат сериализации данных, используемый движком **Source 2** для определения статических свойств сущностей (Entity), таких как Юниты, Герои, Способности и Предметы.

## 2. Принцип работы
Движок игры **автоматически** считывает только задекларированные **KV-файлы**.
Речь идет о файлах, расположенных по стандартным путям (например, ``scripts/npc/npc_units_custom.txt``).

Содержимое этих файлов обрабатывается движком только на **этапе инициализации аддона** (начальный этап перед игрой) и **кэшируется в памяти**, откуда впоследствии считывается сервером и клиентом.
Сервер использует **KV** для создания **Entity** с заданными характеристиками, а Клиент — для отображения свойств в UI и детерминированных вычислений (предикшн).    

### 2.1 Технические требования
Движок игры требует использования кодировки **UTF-8 без BOM** для корректной загрузки.
Использование иных кодировок (например, **UTF-16**) может привести к некорректному чтению данных или критической ошибке парсинга.
Внутри каждого **KV-файла** должен находиться только один корневой объект (таблица), имя которого может быть произвольным.
Кастомные **KV-файлы** могут иметь расширение ``.kv``.

### 2.2 Синтаксис
Синтаксис **KV** структурно схож с **JSON**, но не использует запятые и двоеточия.

```kv3
// Комментарии пишутся через двойной слеш.
"RootName"
{
	"KeyNumber1" 	"10"
	"KeyNumber2" 	"1.1"
	"KeyNumber3" 	".2"
	"KeyString1" 	"string"
	"KeyString2" 	"DOTA_ABILITY_BEHAVIOR_HIDDEN"

	// Вложенный блок.
	"SubBlock"
	{
		"InnerKey"  "InnerValue"
	}
}
```
<Aside type="danger" title="Критично">
  	Нарушение синтаксиса (пропуск символа, незакрытая скобка или опечатка) приводит к тому, что код, идущий после ошибки, игнорируется.
	При этом уведомление об ошибке в консоль **не выводится**.
</Aside>

#### Типы значений
* ``"10"`` — Целое число (number).
* ``"0.1"`` — Число с плавающей точкой. Движок может представить его как *~0.10000000149012* из-за погрешности **floating-point precision**.
* ``".2"`` — Сокращенная запись float-числа (эквивалентно ``0.2``).
* ``{ ... }`` — Вложенная таблица со своими ключами и значениями.

#### Особенности
Помимо строк и чисел, поля типа **AbilityBehavior** (флаги) принимают строку, в которой можно указать множество значений через разделитель `` | ``.    
Например:
```
"AbilityBehavior" "DOTA_ABILITY_BEHAVIOR_HIDDEN | DOTA_ABILITY_BEHAVIOR_NO_TARGET | DOTA_ABILITY_BEHAVIOR_TOGGLE"
```
Хотя технически это строка, парсер при создании экземпляра способности преобразует её в битовую маску.

### 2.3 Runtime перезагрузка
После загрузки **KV-данные** становятся статичными для C++ части движка.
Однако с помощью API метода ``GameRules:Playtesting_UpdateAddOnKeyValues()`` можно инициировать повторный парсинг всех задекларированных KV-файлов во время работы аддона без перезапуска.

<Aside type="caution" title="Предупреждение">
	Метод может некоректно обрабатывать сущности с ``include_keys_from`` при повторной загрузке.
	И при оверрайде стандартных entity.
	В таких случаях рекомендуется просто перезапустить аддон (только кстомку, не сами тулзы). 
</Aside>

## 3. Модульность
С помощью директивы препроцессора ``#base`` можно импортировать ключи из внешнего файла в текущий.
При этом, если ключи в подключаемом файле совпадают с ключами в текущем, **импортируемые значения не перезапишут исходные**.
Приоритет остается за текущим файлом.

#### Пример
``table.txt:``
```kv3
#base "table_2.txt"

"Root"
{
	"key_1" "1"
	"key_2" "2"
}
```

``table_2.txt:``
```kv3
"Root"
{
	"key_2" "0"
	"key_3" "3"
}
```

``Итог при считывании:``
```kv3
"Root"
{
	"key_1" "1"
	"key_2" "2" // Значение осталось "2", так как table.txt имеет приоритет
	"key_3" "3"
}
```

При использовании ``#base`` имя корневого объекта (Root) в подключаемом файле не обязательно должно совпадать с текущим. Директива работает исключительно с содержимым (ключами) внутри блоков.
Директиву ``#base`` можно указывать как до, так и после объявления корневого объекта — разницы в поведении не обнаружено.

## 4. Наследование
Специальный ключ ``include_keys_from`` позволяет *скопировать* все ключи из другой, **уже определенной ранее** KV-сущности.
Всегда следите за порядком декларации: Entity, используемые в качестве базовых, должны быть занесены в память раньше наследников.

## 5. Примеры использования

### 5.1 Пайплайн создания юнитов
Рекомендуемый подход к архитектуре **KV-юнитов**: создание "базового" юнита с описанием всех стандартных ключей, создание категорий на его основе, и, наконец, создание конкретных юнитов на основе категорий.
Этот метод наследования (от общего к частному) избавляет от дублирования кода.

Пример иерархии:
```kv
"npc_unit_base"
{
	... // Описываем все общие ключи и значения
}

"npc_unit_neutral_creep"
{
	"include_keys_from" "npc_unit_base"

	...
}

"npc_unit_neutral_wolf"
{
	"include_keys_from" "npc_unit_neutral_creep"
	"LocToken"			"npc_unit_neutral_wolf"

	...
}

"npc_unit_neutral_wolf_2"
{
	"include_keys_from" "npc_unit_neutral_wolf"

	...
}

"npc_unit_line_creep"
{
	"include_keys_from" "npc_unit_base"

	...
}

"npc_unit_line_creep_melee"
{
	"include_keys_from" "npc_unit_line_creep"
	"LocToken"			"npc_unit_line_creep_melee"

	...	
}
```
Важно учитывать: при использовании ``include_keys_from``, параметры локализации (имя юнита в UI) также наследуются от родителя.
Для явного указания ключа локализации конкретного юнита используйте параметр ``LocToken``.

### 5.2 Оверрайд стандартных entity
Архитектура игры позволяет изменять параметры стандартных героев, способностей и предметов без необходимости создавать их полные копии.
Этот процесс называется оверрайдом (переопределением).
Движок сначала загружает стандартные файлы игры, а затем *накладывает* на них ваши изменения из файлов *_custom.txt.
Если ключи совпадают, значение из вашего файла имеет приоритет. Например:    

``npc_units_custom.txt``:
```kv
"DOTAUnits"
{
	"npc_dota_unit_undying_zombie" // Стандартный зомбик
	{
		"ModelScale"	"0.5"
		// Теперь в кастомке он будет в 2 раза меньше, а все базовые свойства сохранятся.
	}
}
```
По такой же аналогии можно переопределить **KV-данные** всех базовых **entity**.

``npc_items_custom.txt``:
```kv
"DOTAAbilities"
{
    "item_blink"
    {
        "AbilityCastRange"				"200"
    }
}
```

### 5.3 Рекомендации
Чтобы понять какие ключи и значения у **entity** существуют, можно обратиться к базовым **entity** **Dota 2**.
Все их **KV-данные** описаны в основном vpk архиве игры по пути ``scripts/npc/...``.

Список файлов для ознакомления:
* ``npc_abilities.txt`` — Все способности героев и крипов.
* ``npc_heroes.txt`` — Параметры героев.
* ``npc_units.txt`` — Параметры обычных юнитов, башен и вардов.
* ``items.txt`` — Предметы.
